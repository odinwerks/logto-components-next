# Plan: Fix RBAC Organization Roles Using OIDC Claims

## Root Cause
**M2M applications cannot access organization-specific Management API endpoints** because organization operations require user context. The current implementation incorrectly uses Management API (`/api/organizations/{id}/users/{userId}/roles`) with an M2M token to fetch roles, resulting in `ERR_JWT_CLAIM_VALIDATION_FAILED`.

## Solution
Use the `organization_roles` claim from the OIDC `/oidc/me` endpoint, which is included when the `urn:logto:scope:organization_roles` scope is requested. The claim format is: `["organization-id:role-name"]`.

## Implementation Steps

### 1. Create parser for organization_roles claim
- Add `parseOrganizationRoles()` function in `validation.ts` or new utility
- Transform `string[]` of `"orgId:roleName"` into `Array<{id: string, name: string, organizationId: string}>`
- Example: `"gov-123:president"` → `{id: "gov-123:president", name: "president", organizationId: "gov-123"}`

### 2. Update fetchDashboardData in actions.ts
- Remove the call to `fetchOrganizationRoles()` (lines 203-207)
- Parse `userInfo.organization_roles` directly from the OIDC response
- Map to expected `UserData.organizationRoles` format
- Add the scope must be enabled in Logto Console (already documented)

### 3. Remove Management API dependency for roles
- Delete or deprecate `fetchOrgRoles()` and `fetchOrganizationRoles()` functions
- Keep `getManagementApiToken()` for other operations (user deletion, etc.) that legitimately need it

### 4. Verify Protected component
- The server-side `Protected` component in `custom-logic/Protected.tsx` currently uses `validateRbac()` which calls `checkRoleInOrg()` → `fetchOrgRoles()`.
- Must update `validateRbac()` to accept pre-pared organization roles instead of fetching via API.
- Pass parsed roles from `fetchDashboardData` to the Protected component's server actions.

### 5. Update ClientProtected alignment
- ClientProtected already expects `userData.organizationRoles` array.
- No changes needed if data format matches.

## Configuration Requirements
- Logto Console: Custom JWT → Toggle ON `organization_roles` claim
- App scopes must include: `urn:logto:scope:organization_roles` (already in `.env.example` and README)

## Expected Format
```typescript
// Input from OIDC:
userInfo.organization_roles = ["org-abc:admin", "org-xyz:member"]

// Parsed output:
organizationRoles: [
  { id: "org-abc:admin", name: "admin", organizationId: "org-abc" },
  { id: "org-xyz:member", name: "member", organizationId: "org-xyz" }
]
```
